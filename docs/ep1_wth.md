# Episode 1: First hit on a wall: Wth I'm REALLY doing?

I knew this would not be easy. Like, if creating programming languages were easy there would be a lot of them out there, right? People would have been using AI to make them just for the sake of it, as they do with web pages and so. But even thought AI is theorically capable of achieving such task, why people don't go ahead with it?
Sure you can say "because there is no point in creating a new language when we able a lot already", but wouldn't that be the case for any software? At this point of computing history we have software for almost everything, yet we still create new ones, with more features or aesthetics. Kinda like art. **Why to keep making new paintings when there are a lot already?** I hope you get my point.

The thing is that I knew this wouldn't be easy, but more than it, is the fact that making such big project for the first time feels like walking in a dark room. By the time I started this project I thought that a definition of the project, syntax and such would be "the time consuming" part, and that the rest would be just coding: loading a file, parsing, creating a Tree<Token> or so. Not need to say how wrong I was.
I hit my head against the keyboard for ten hours straight, two days in a row to realize **I was not ready for this** and I started to heavily doubt about what in the world I was doing. Is a new programming language really needed? Not too long I found the answer, the one that really motivated me to start all this:

> I didn't want to create this for the sake of bringing to the world something new or revolutionary. I wanted to write my own programming language for the sake of myself: I wanted to learn more, to enjoy such difficult, yet rewarding task. ~~Also because I wanted a cute sounding programming language.~~

I know we code for work, because the bills need to be paid and because we can help our momma with our income. But I love computers, let me be romantic about it. Let me think that **programming like such other art, must be enjoyed.**

I'm far from being smart in these architectural matters, and so I asked the AI for resources to build compilers: no videos, no short tutorials. The AI then suggested what I think is an amazing resource so far (I literally read over 100 pages in a sit, but more of that later on), which is a book called Crafting Interpreters, by Robert Nystrom.
I checked it on Amazon and it was E-X-P-E-N-S-I-V-E. I'm not from a high income country, I speak Spanish, ya know?

Jokes aside, I'm still a student and in fact it is true that I have no money to spend in non-vital things. The book was expensive for my budget and I had to be really conscious about this purchase. And so I ended up buying this book and **making myself the promise to take the most of it**. Surprisingly, the book was way more friendly that I thought (I was scared of finding that I paid for something I can't really understand), and I liked it so much that I end up reading the whole day long.

I decided to stick to what the book suggests in terms of what language to create and how, which is called lox. The book is divided in two sections, where in the first we create a lox compiled using java (jlox), and in the second one, using C (clox). Sadly for myself, **I won't be able to continue mei development until I finally grasp these concepts**. I need to understand a lexer, how to build and interpret a Syntax Tree, decide whether mei will have optimizations or not, and so on.

I will work my way through the book, which will take some time (I expect around 3 to 4 months). Thought, I will keep updating this repository with my notes and learning while crafting jlox and clox compilers, as this journey needs to be documented.
See you in the next episode about lexers!
